# 贝叶斯修改意见

## 一句话结论

当前 `Bayesianpolicy-lite.md` 的大方向是对的：它已经从“用 C_hat 手动调权重”的 heuristic policy，推进到了“学习 P(outcome | task_family, action)，再用后验期望价值推导策略倾向”的 Bayesian-inspired policy。

但是，当前版本仍然有一个重要问题：

```text
它口头上说 rule 是 fallback，
但公式上 rule 仍然参与主策略混合。
```

如果我们担心 rule 没有充分文献依据，那么建议把主线从：

```text
rule-heavy fallback + Bayesian posterior mixing
```

改成：

```text
weak prior + Bayesian posterior
rule 只做 safety guard
```

这会让机制更干净，也更容易向 AAAI / AISI 审稿人解释。

## 1. 当前版本的问题在哪里

当前文档里的核心混合公式是：

```text
pi_base(a | s) = (1 - c) * pi_rule(a | s) + c * pi_Bayes(a | f)
```

这个公式的问题是：当经验较少、`c` 较低时，系统主要依赖 `pi_rule`。

也就是说：

```text
冷启动阶段：rule 主导
经验增加后：Bayesian posterior 逐渐主导
```

如果 `pi_rule` 本身是一组手写权重，例如：

```text
helplessness 高 -> attempt_self 降低
support_quality 高 -> seek_help 增加
consecutive_failures 多 -> avoid 增加
```

审稿人会追问：

```text
这些权重从哪里来？
为什么是这个阈值？
为什么加 0.10 而不是 0.20？
这些 rule 是否主导了实验结果？
```

所以，如果我们希望 Bayesian policy-lite 更像一个清晰的机制贡献，就不应该让 rule 长期参与主策略概率。

## 2. 建议改成 weak prior + Bayesian posterior

建议把主公式改为：

```text
pi_base(a | f) = (1 - c) * pi_prior(a | f) + c * pi_Bayes(a | f)
```

其中：

```text
pi_prior = 很弱的初始先验
pi_Bayes = 从过去 action-outcome 经验中学出来的后验策略
c = posterior evidence confidence
```

这样解释会更清楚：

```text
没有经验时：agent 依赖一个很弱的默认先验
有经验后：agent 主要依赖自己的历史 action-outcome 经验
rule：只负责异常保护，不再作为主机制
```

这比 `pi_rule` 混合更像 Bayesian 机制。

## 3. pi_prior 应该是什么

第一版不建议设计复杂的 rule prior。越复杂，越容易被质疑。

推荐使用很弱的中性先验：

```text
attempt_self: 0.34
seek_help_then_attempt: 0.33
avoid: 0.33
```

或者使用非常轻量的 task-risk prior：

```text
低风险任务：
attempt_self 稍高

高风险任务：
seek_help_then_attempt 或 avoid 稍高
```

但这个调整必须很小，例如：

```text
单个 action 的 prior 调整不超过 0.05 或 0.10
```

更稳妥的第一版是直接用 neutral prior，因为它最容易解释：

```text
我们不把人工规则写进策略主机制，而是让经验后验逐渐决定策略。
```

## 4. rule 应该保留，但只做 safety guard

不建议完全删除 rule。它仍然可以有用，但角色要改。

建议 rule 只做这些事情：

```text
1. invalid posterior fallback
2. probability clipping
3. extreme risk guard
4. impossible action guard
5. debugging baseline
```

也就是说，rule 不再参与常规策略混合：

```text
不推荐：
pi_base = (1-c)pi_rule + c pi_Bayes
```

推荐：

```text
pi_base = (1-c)pi_prior + c pi_Bayes
pi_final = safety_guard(pi_base)
```

这里的 `safety_guard` 只在极端情况触发，例如：

```text
posterior 无效
概率全是 0
任务风险极高
action 不适用于当前任务
```

这样 rule 就不再是理论主线，而是工程保护层。

## 5. confidence 需要按 action 分开

当前文档里写的是：

```text
c = min(1, total_updates_for_task_family / K)
```

这个太粗。

问题是：agent 可能在某个任务类型里大量 `avoid`，但几乎没有 `attempt_self`。这时总经验很多，但并不代表它真的知道 `attempt_self` 会怎样。

更合理的是：

```text
c_a = confidence(task_family, action)
```

例如：

```text
c_attempt_self = min(1, updates(f, attempt_self) / K)
c_seek_help = min(1, updates(f, seek_help_then_attempt) / K)
c_avoid = min(1, updates(f, avoid) / K)
```

再用 action-specific confidence 混合：

```text
score(a) = (1 - c_a) * pi_prior(a | f) + c_a * pi_Bayes(a | f)
pi_base = normalize(score)
```

这样模型不会“假装知道自己没试过的 action”。

## 6. posterior 应该学什么

建议保留当前文档的核心：

```text
P(outcome | task_family, action)
```

用 Dirichlet posterior 维护每个任务类型、每个 action 下的 outcome 分布：

```text
memory[task_family][action].alpha[outcome]
```

比如：

```text
payment_risk_confirmation / attempt_self:
  success_self
  failure_after_attempt
  abandon_midway

payment_risk_confirmation / seek_help_then_attempt:
  success_with_help
  failure_even_with_help
  abandon_midway

payment_risk_confirmation / avoid:
  avoid_without_attempt
```

关键不是只记录“成功率”，而是记录：

```text
不同 action 分别导致什么 outcome
```

这才更接近 Bayesian behavioral control 论文里的 action-outcome prediction。

## 7. utility 要诚实写成 theory-grounded design choice

Bayesian policy-lite 里有两部分：

```text
1. posterior belief：从经验中更新
2. utility mapping：研究者根据理论设定
```

审稿人可能接受第二点，但不能把它包装成“模型自己学出来的”。

建议论文里明确写：

```text
The posterior over action-conditioned outcomes is learned from simulated experience,
whereas the outcome utility mapping is theory-grounded and evaluated through sensitivity analysis.
```

通俗说：

```text
概率是 agent 从经验里学的；
每种结果有多好或多坏，是我们基于文献和理论设定的；
为了防止手调，我们会做敏感性分析。
```

## 8. shadow mode 应该是第一阶段

不建议一开始就让 Bayesian policy-lite 真正控制 agent 行为。

最近一周最稳的路线是：

```text
shadow mode
```

意思是：

```text
真实行为仍然按当前主实验机制运行；
Bayesian policy-lite 在旁边计算 pi_Bayes、Q_Bayes、confidence；
所有结果写入 payload，供后续分析。
```

这样可以回答：

```text
Bayesian posterior policy 是否能解释 agent 后续的求助、尝试和回避？
Bayesian posterior policy 是否比 rule 更能预测失败、放弃或求助成功？
Bayesian posterior policy 是否和 helplessness / felt control / avoid_reason 有一致关系？
```

如果 shadow mode 结果不好，不会破坏主实验。

如果 shadow mode 结果好，再考虑 gated mode。

## 9. 推荐的实现模式

建议文档中加入四种模式：

```text
off:
  不启用 Bayesian policy-lite

shadow:
  只计算 pi_Bayes，不影响真实 action

gated:
  当 evidence confidence 足够时，小幅混入 pi_Bayes

on:
  Bayesian posterior policy 成为主策略来源
```

近期只建议做：

```text
off + shadow
```

暂时不要做：

```text
on
```

因为 on 会改变主实验机制，使结果解释变复杂。

## 10. 和 LLM simulation 的关系

Bayesian policy-lite 不应该替代 LLM simulation。

建议定位为：

```text
LLM simulation 是主体；
Bayesian posterior policy 是可审计的 action-outcome 学习插件；
rule 是安全保护层；
utility 是理论设定；
shadow mode 是验证机制合理性的第一步。
```

也就是说：

```text
LLM 负责人物语义、任务理解、访谈、归因、叙事一致性；
Bayesian 模块负责记录过去 action-outcome 经验，并估计下次不同 action 的结果概率；
rule 不再负责解释行为，只负责异常兜底。
```

推荐论文定位：

```text
LLM-based agent simulation with a lightweight Bayesian-inspired posterior predictive action-outcome module.
```

不推荐定位：

```text
Bayesian RL agent simulation.
```

## 11. 建议修改 Bayesianpolicy-lite.md 的地方

建议把第 9 节标题从：

```text
Rule 作为 fallback，而不是主线
```

改成：

```text
Weak prior 作为 cold-start，rule 只做 safety guard
```

建议把公式从：

```text
pi_base(a | s) = (1 - c) * pi_rule(a | s) + c * pi_Bayes(a | f)
```

改成：

```text
raw_score(a | f) = (1 - c_a) * pi_prior(a | f) + c_a * pi_Bayes(a | f)
pi_base(a | f) = normalize(raw_score)
pi_final = safety_guard(pi_base)
```

建议把决策链从：

```text
state
-> pi_rule fallback
-> posterior P(outcome | task_family, action)
-> Q_Bayes
-> pi_Bayes
-> pi_base = (1-c)pi_rule + c pi_Bayes
-> optional bounded LLM adjustment
-> pi_final
```

改成：

```text
state / task_family
-> weak pi_prior
-> posterior P(outcome | task_family, action)
-> Q_Bayes
-> pi_Bayes
-> action-specific confidence c_a
-> pi_base = normalize((1-c_a)pi_prior + c_a pi_Bayes)
-> optional bounded LLM semantic adjustment
-> safety guard
-> pi_final
```

如果是 shadow mode，则最后一步不影响真实 action，只记录：

```text
pi_prior
pi_Bayes
q_Bayes
confidence_by_action
pi_base_shadow
pi_final_shadow
actual_strategy
```

## 12. 建议新增的 payload 字段

建议 shadow mode 在每次 episode 写入：

```json
{
  "bayesian_policy_lite": {
    "version": "policy_lite_v1",
    "mode": "shadow",
    "task_family": "payment_risk_confirmation",
    "pi_prior": {
      "attempt_self": 0.34,
      "seek_help_then_attempt": 0.33,
      "avoid": 0.33
    },
    "q_bayes": {
      "attempt_self": -0.20,
      "seek_help_then_attempt": 0.55,
      "avoid": -0.05
    },
    "pi_bayes": {
      "attempt_self": 0.22,
      "seek_help_then_attempt": 0.60,
      "avoid": 0.18
    },
    "confidence_by_action": {
      "attempt_self": 0.50,
      "seek_help_then_attempt": 0.67,
      "avoid": 0.17
    },
    "pi_shadow": {
      "attempt_self": 0.28,
      "seek_help_then_attempt": 0.51,
      "avoid": 0.21
    },
    "actual_strategy": "attempt_self",
    "status": "shadow_only"
  }
}
```

这个字段非常重要，因为后续可以直接做分析：

```text
Bayesian shadow policy 和真实 action 是否一致？
Bayesian shadow policy 是否预测了后续失败或求助？
不同 friction 条件下 pi_Bayes 是否系统性变化？
```

## 13. 本周建议完成什么

建议本周目标不要太大。

优先级最高：

```text
1. 修改设计文档口径：weak prior + Bayesian posterior + safety guard
2. 实现 shadow mode，不影响真实行为
3. 新增 proto_bayesian_policy_memory
4. 在 payload 中记录 pi_prior / pi_Bayes / q_Bayes / confidence_by_action / pi_shadow
5. 增加最小测试，确认 shadow mode 不改变原策略选择
```

暂时不建议：

```text
1. 直接 on mode
2. 删除现有 rule 系统
3. 把它写成 full Bayesian RL
4. 大改 LLM strategy deliberation
```

## 14. 审稿人视角的最终判断

如果按照当前文档原样实现，审稿人可能会说：

```text
你们虽然有 posterior，但 rule 仍然在主公式里。
如果结果好，无法判断是 Bayesian posterior 有效，还是人工 rule 起作用。
```

如果改成 weak prior + Bayesian posterior + safety guard，审稿人更容易接受：

```text
冷启动由弱先验处理；
行为适应主要来自 action-conditioned outcome posterior；
rule 不承担理论解释，只承担工程保护；
LLM simulation 的主体仍然保留。
```

因此，推荐修改方向是：

```text
从 rule-mixed Bayesian policy-lite
改成 weak-prior Bayesian posterior predictive policy-lite
```

这条路线最适合你们当前实验，也最不容易把系统变成纯 RL agent。
