# 2026-04-23 Meeting Notes

## 一、会议主题

本次会议主要围绕当前 digital friction 实验中的 helplessness 机制形式化展开，重点讨论了：

- attribution 中的 scope 如何定义与形式化；
- scope spillover 是否可以使用 Gaussian / similarity-based generalization；
- task family 应该采用医院场景还是泛数字任务场景；
- similarity matrix 如何从人工设定走向 NLP 语义相似度；
- Bayesian behavioral control 论文如何接入当前 controllability / helplessness 更新主线；
- 当前模型如何验证、如何设置 baseline、如何解释参数。

## 二、关于 attribution scope 的讨论

当前 attribution reformulation 中主要有三个维度：

- locus
- stability
- scope

其中 locus 相对更容易找到形式化或相关计算建模支撑，而 stability 和 scope 都缺少成熟、细化的形式化更新方程。因此当前先从 scope 入手是可行的。

老师强调，scope 需要先给出一个清晰定义。它不能只是抽象地说“范围”，而要说明一次失败被个体解释为：

- 只是当前这个具体任务不行；
- 还是这一类相似任务都不行；
- 甚至更广泛地认为自己在数字任务中都不太行。

因此，scope 更接近于 failure attribution 的泛化范围，而不是本次失败本身有多严重。

一个直观例子是：如果用户微信支付失败，可能不只是抗拒继续用微信支付，也可能对支付宝支付、线上付款等相似任务产生回避。这种影响不是“发生或不发生”的概率，而是扩散强度。

## 三、关于 scope spillover 与 Gaussian 形式化

当前方案中，scope 不直接更新 helplessness 主变量，而是影响其他 task family 的 self-efficacy。老师认为这个方向是可行的。

当前核心含义可以理解为：

一次任务失败后，如果 LLM 判断这次失败具有较强 scope attribution，那么它不只影响当前任务，还会按任务相似度影响其他 task family 的 self-efficacy。越相似的任务受到影响越强，越不相似的任务受到影响越弱。

老师对公式表达提出了几个建议：

- 主公式应该放在前面，再解释其中的子项。
- 公式介绍应当从“大公式”到“小组件”，不要先讲距离、权重，最后才讲更新量。
- 最后一行 self-efficacy spillover 更新才是核心公式，应放到更突出的位置。
- `spill` 表示 spillover，即扩散影响。
- 需要明确 `i` 和 `j` 的含义，例如 `i` 是当前失败发生的 source task family，`j` 是被影响的 target task family。
- `G_t` 这类从泛化公式里抄来的符号如果不能解释清楚，应删除。
- `beta` 如果只是整体缩放参数，可以先固定为 1，或者暂时弱化它的解释。
- `sigma` 与 similarity matrix 都会影响扩散范围，两者关系需要讲清楚，第一版可以先固定 `sigma`。

老师认为 Gaussian / 距离衰减方法本身可以作为一种借鉴，但在汇报时不需要把泛化文献讲得太复杂。只需要用一两句话说明：

泛化机制指的是，人会把一次经验推广到相似情境中；这种推广强度通常会随着情境相似度下降而减弱。我们借用的是这种“相似性越高，影响越强”的形式化写法。

## 四、关于 task family 设计

老师指出，当前任务设计存在一个问题：任务抽象层级不够一致。

例如当前的四类任务中：

- 登录验证是比较泛的数字功能；
- 支付结算也是比较泛的数字功能；
- 挂号预约和健康信息查询则明显带有医院场景；
- 因此整体上会让人困惑：研究到底是在做医院数字服务场景，还是泛数字任务场景？

老师建议必须二选一：

### 方案 A：具体场景

如果选择医院场景，就把所有任务都放在医院数字服务里，例如：

- 医院 App 登录验证；
- 在线挂号预约；
- 医疗费用支付；
- 检查报告或健康信息查询。

这样好处是场景具体，读者容易想象，描述也不会发散。

### 方案 B：泛数字任务

如果选择 general digital task，就要把任务全部改成更通用的底层数字功能，例如：

- 登录 / 注册 / 身份验证；
- 信息查询；
- 个人信息修改或账户资料管理；
- 表单填写与提交；
- 支付结算。

这样好处是更泛化，但需要任务定义更清楚，不能混入过强的医院场景描述。

老师建议，如果走泛数字任务路线，就可以观察多个常见 App 或电子服务中的共性功能，例如微信、支付宝、美团等都可能包含：

- 登录 / 注册；
- 个人中心；
- 查询信息；
- 提交某个申请或订单；
- 支付；
- 修改个人信息。

这些可以作为泛数字 task family 的来源。

## 五、关于 similarity matrix 的讨论

当前 similarity matrix 是人工设定的，例如任务之间直接给一个相似度值。老师认为在任务数量很少、任务固定的第一版里，这样可以先做。

但如果想让 similarity matrix 更有依据，建议使用 NLP 语义相似度方法。

基本思路是：

1. 给每个 task family 写一个清晰的文本定义。
2. 给每个 task family 补充典型使用场景。
3. 把这些文本转成向量。
4. 用向量之间的距离或 cosine similarity 计算任务相似度。

例如：

- 登录验证有自己的 definition 和使用场景；
- 预约提交也有自己的 definition 和使用场景；
- 将两个任务描述文本输入 embedding 模型后，可以得到两个向量；
- 两个向量越接近，说明任务语义越相似。

老师认为这比完全人工给相似度更有依据，也比较容易实现。

## 六、关于 Bayesian behavioral control 论文

会议中讨论了 Bayesian formulation of behavioral control 这篇论文。

这篇论文使用了类似 multi-armed bandit 的简化环境，讨论个体如何形成 controllability belief / controllability prior。

论文中 control 可以拆成三层理解：

1. 单个动作的结果是否稳定；
2. 不同动作是否能稳定带来不同结果；
3. 那些真正有价值的结果，是否可以通过行为可靠获得。

老师特别提醒：需要回到原文确认核心公式到底对应哪一层 control。不能只凭感觉说它是第三层。如果公式只形式化第三层，那么后续实验解释时就不能随意用第一层、第二层的含义来解释结果。

这件事很重要，因为如果公式和解释层级不一致，后面用实验结果解释心理机制时会不严谨。

目前这篇论文可以被理解为：它没有直接形式化 helplessness，而是形式化更前面的 controllability belief / controllability prior。这个 prior 会影响个体进入新任务后的 action-outcome prediction、探索意愿和行为选择。

老师认为这篇论文和当前研究很接近，也比较“正经”和科学，但它的任务环境非常理想化，不区分年龄、性别和社会背景。因此它可以作为 controllability 更新主线的参考，但不能直接等同于老年数字任务场景。

## 七、关于老年人场景的特殊性

老师补充提到，老年人在数字任务中的 helplessness 可能不只是一般人面对不可控环境的反应，还可能受到社会文化背景影响。

例如，在中国社会语境中，老年人可能被默认认为“不适合做很多数字事情”“学不会新东西”。这种外部社会规约可能内化为老年人的自我认同和自我价值判断。

这个方向目前比较难直接建模，可以作为后续扩展，不一定放入当前第一版系统。

## 八、关于模型验证与 baseline

当前主要困惑是：即使加入更形式化的 scope / controllability 更新，实验结果在定性方向上可能和简单加减规则差不多，那么如何证明新机制更好？

老师建议从两个角度考虑。

### 1. 纯 LLM 作为 baseline

可以把“没有显式公式、完全由 LLM 判断和更新”的版本作为 baseline。

原因是：

- 它没有明确的人类机制约束；
- 不同轮次输出可能波动较大；
- 可能今天给很强的 scope，明天又给很弱的 scope；
- 缺乏稳定、可解释的状态转移机制。

相比之下，当前方法的优势不是简单提高百分比，而是把心理机制拆成明确状态变量和更新方程，使每个参数有心理或行为含义。

### 2. 需要寻找 metric

老师指出，目前最缺的是一个可以衡量模拟是否“像真实情况”的 metric。

如果未来能找到真实数据或合理指标，就可以比较：

- 纯 LLM baseline；
- 简单加减规则；
- 当前带公式约束的机制模型；
- 去掉某个模块后的 ablation。

如果加入某个模块后，模拟结果更接近真实数据，就能说明这个机制有价值。

老师也提到，如果未来能够通过真实数据拟合出某个扩散参数，例如扩散强度大约是多少，那么这本身也可以成为一个研究贡献：过去文献只说 helplessness 会泛化或扩散，但没有给出具体量化强度；我们的模型可以在特定场景下给出一个可参考的量化结果。

## 九、关于参数解释

老师强调，每一个参数都必须有清晰的心理或行为含义。

例如：

- scope amplitude 表示这次失败被归因为更广范围问题的强度；
- similarity 表示两个 task family 之间的相似程度；
- sigma 表示扩散影响随距离衰减的范围；
- beta 如果存在，就只能解释为整体缩放强度。

如果某个参数只是为了调结果而存在，就会削弱模型的科学性。因此后续加参数前要先想清楚它对应什么心理含义。

老师也提到，参数值本身未必一开始就能有真实世界的绝对解释。更现实的做法是：

- 先明确参数含义；
- 再通过实验、ablation 或真实数据拟合，观察什么参数范围能产生更合理的行为；
- 最后再解释该参数在当前场景中的含义。

## 十、下一步任务

本次会议后明确的任务包括：

1. 重新梳理 scope 的定义，讲清楚它是 failure attribution 的泛化范围，而不是失败严重程度。
2. 调整 scope 公式展示顺序，将主更新公式放在前面，再解释 `A_t`、similarity、Gaussian weight 等子项。
3. 删除或解释不清楚的符号，例如 `G_t`。
4. 决定任务设计路线：医院数字服务场景，或泛数字任务场景。
5. 如果选择泛数字任务，需要把任务统一改成更 general 的数字底层功能。
6. 尝试使用 NLP embedding 方法计算 task family 的 semantic similarity。
7. 回到 Bayesian behavioral control 原文，确认核心公式到底对应 control 三层中的哪一层。
8. 思考如何将 Bayesian controllability prior 接入当前 helplessness 更新主线。
9. 设计 baseline，至少包括纯 LLM baseline。
10. 寻找可用于评估模拟真实性的 metric 或真实数据来源。
11. 后续可以考虑 intervention intensity、帮助时机、帮助比例等作为实验变量。
